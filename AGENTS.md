# Agent Instructions

## 新会话默认入口 / New Chat Defaults

新开的 Codex 对话进入本仓库时，除非用户明确要求跳过，应优先读取：
- `README.md` 或 `README.zh-CN.md`
- `docs/PROJECT_MAP.md`
- `docs/COMMANDS.md`
- 与当前任务相关的 `docs/HANDOFFS/*.md`

读取这些文档后，再根据当前任务只阅读相关源码文件；不要为了建立上下文而默认扫描整个仓库。

如果用户只给出一个简短任务，可以按 `docs/NEW_CHAT_PROMPT.md` 的模板理解任务入口。
如果用户提到提交、push、commit、GitHub 或 PR，优先读取 `docs/GIT_WORKFLOW.md`。

## 并行开发规则 / Parallel Development

本仓库可能会被多个 Codex 对话并行开发。

开始修改前：
- 先检查当前 git 状态；如果当前目录不是 git 仓库，请说明这一点。
- 先阅读相关文件再编辑。
- 默认认为工作区里可能已有用户或其他 Codex 对话的改动。
- 不要覆盖无关改动。
- 修改范围必须限制在本次任务内。
- 避免进行与任务无关的大范围重构、格式化或依赖升级。

## 禁止默认修改的文件 / Protected Files

除非用户明确要求，不要修改：
- `.env`
- `.env.local`
- `package-lock.json`
- `pnpm-lock.yaml`
- `yarn.lock`
- 部署配置文件
- 全局项目配置文件
- 其他明显与当前任务无关的共享配置文件

## 任务范围 / Task Scope

每次任务都应优先遵守用户在当前对话中给出的范围。

如果任务范围不明确：
- 先根据现有文件和上下文做保守判断。
- 只修改完成任务所必需的文件。
- 如果继续操作可能覆盖他人改动或引入较大风险，请先说明风险并等待确认。

## 检查线程 / Review Threads

如果当前任务是检查、审查或 review：
- 默认只做 review，不修改代码。
- 优先指出 bug、回归风险、缺失测试、不一致实现和集成风险。
- 发现问题时给出文件路径和具体原因。
- 只有在用户明确要求修复时才编辑文件。

## 实现线程 / Implementation Threads

如果当前任务是实现功能或修复问题：
- 保持改动聚焦，不扩散到无关模块。
- 复用项目已有风格、框架和工具。
- 不新增不必要的抽象。
- 涉及用户可见行为时，尽量进行可验证的本地检查。

## 整合线程 / Integration Threads

如果当前任务是整合多个对话或多个改动：
- 先梳理每个线程的任务范围和改动结果。
- 检查是否存在文件冲突、重复实现或接口不一致。
- 优先保证当前工作区状态可理解、可验证、可继续开发。
- 在整合前后说明关键取舍。

## 完成时交接 / Handoff Summary

每次任务结束时，请输出交接摘要，包括：
- 修改了哪些文件
- 实现了什么
- 没有改什么
- 运行了哪些命令或测试
- 验证结果
- 已知风险
- 下一步建议
